\section*{1. Problem Statement}

The prediction of the stock market remains one of the most persistent challenges in quantitative finance. Traditional financial analysis relies heavily on the Global Industry Classification Standard (GICS), a hierarchical framework classifying over 43,000 publicly traded companies into 11 broad sectors such as "Technology" and "Energy."\\
The GICS, however, rests on the assumption that companies within the same category exhibit similar price behaviors, while companies across categories behave differently. In practice, stock market value is driven by geopolitical and socioeconomic events, making consistently reliable predictions exceptionally difficult. A single major economic event can simultaneously affect multiple, seemingly unrelated industries through higher-order dependencies.\\
\textbf{Our question:} \textbf{To what extent do traditional GICS sectors remain distinct to one another, and can graph-based detection identify 'pivot stocks' that connect supposedly unrelated industries?}

\subsection*{Dataset(s) Used}
This project's dataset is a subset of Yahoo's financial data\textsuperscript{1}. In particular, we use the S\&P 100 companies when building the correlation graph. When the ticker data is downloaded, it returns in the form of a \texttt{pandas.DataFrame}. For more than one ticker, the columns are arranged as a multi-level index (price fields and ticker symbols); we flatten this in \texttt{download\_tickers} in \texttt{data\_reader.py} using \texttt{DataFrame.stack()} and \texttt{DataFrame.reset\_index()} so that each row is one date and one ticker. The flattened table includes \texttt{Date}, \texttt{Ticker}, and OHLCV columns, where OHLCV stands for the usual daily fields: \textbf{O}pen (first trade price), \textbf{H}igh, \textbf{L}ow, \textbf{C}lose (last trade price), and \textbf{V}olume (shares traded). Table~\ref{tab:flattened-prices} shows the shape of that row-wise table with illustrative values; our program only reads \texttt{Date}, \texttt{Ticker}, and \texttt{Close} when we call \texttt{log\_return\_list()} in \texttt{compute.py} to build the log-return series. Under \texttt{yfinance}'s default \texttt{auto\_adjust=True}, the \texttt{Close} column is adjusted for splits and dividends in line with the library's usual behavior. GICS sectors are not columns in that price table. We call \texttt{get\_sectors\_for\_tickers()} separately, which uses \texttt{Ticker.info.get('sector', 'Unknown')} for each ticker, and we attach those sector strings to nodes in the \texttt{CorrelationGraph}.

\section*{Computational Overview}
As outlined in Section 1, we retrieve stock price data from the Yahoo Finance Python API, \texttt{yfinance}. With each ticker represented as a node in a graph, edges are formed based on the correlation between two nodes' price trends. If the strength of the correlation exceeds a user-specified threshold, an edge is formed between them. This forms an undirected weighted graph, which is central in analyzing price behaviors across sectors, such as whether an economic event affecting one sector propagates to another.\\

The overall computation pipeline accepts a user-specified list of thresholds, resulting in a graph being constructed for each threshold. However, the underlying price data is downloaded only once.\\
To construct a graph, the historical price data of the user-specified set of tickers is first downloaded with two additional user-specified parameters: either a start and end date or a time period, and the sampling frequency. Once downloaded, the resulting \texttt{pandas.DataFrame} is then restructured so that the \texttt{DataFrame} can be indexed by ticker and date. We then call \texttt{get\_sectors\_for\_tickers()} with the ticker set to retrieve the sector of each company, and calculate the log returns, the natural logarithm of the ratio of consecutive stock prices, by calling \texttt{log\_return\_list()} with our DataFrame. Doing so produces one dictionary with the ticker names and corresponding GICS sectors, and another dictionary which pairs tickers to their respective list of log returns. We then add each node, which has attributes \texttt{ticker}, \texttt{sector}, and \texttt{neighbours}, into a \texttt{CorrelationGraph} object, forming the underlying graph structure. Edges are added by computing the Pearson Correlation Coefficient between each pair of nodes using their lists of log returns. The value is then compared to a user-specified threshold. An edge is added between two nodes if the absolute value of the Pearson coefficient is greater than the threshold to identify sufficiently strong positive and negative correlations.\\\\

The module used for data retrieval is \texttt{yfinance-cache}, an unofficial caching wrapper of \texttt{yfinance} designed to prevent rate-limiting by the Yahoo Finance API by caching previously retrieved data locally. Data is retrieved by calling the \texttt{yfinance\_cache.download()} function. A \texttt{yfinance\_cache.Ticker} object, instantiated with a ticker symbol, contains all available information for that stock. Calling \texttt{Ticker.info.get('sector', 'Unknown')} returns the sector of each ticker.\\
The module used for data storage and manipulation is \texttt{pandas}, a library providing high-performance data structures and tools to manipulate tabular data. Data retrieved from  \texttt{yfinance\_cache.download()} is returned as a \texttt{pandas.DataFrame} object. The \texttt{DataFrame} is then flattened from a multi-level column format to a row-wise format indexed by date and ticker by calling \texttt{DataFrame.stack()} and \texttt{DataFrame.reset\_index()}. This format allows direct access to the close prices per ticker and date from which log returns are calculated.

We also implement the crash simulation and pivot-candidate ideas from our proposal inside \texttt{CorrelationGraph}. The function \texttt{bfs\_crash\_simulation(start\_ticker)} runs a breadth-first search along correlation edges from a user-selected ``crashing'' stock, recording nodes layer by layer up to a maximum depth given by \texttt{BFS\_DEPTH} in \texttt{constants.py}. The helper \texttt{\_reachable\_from\_crash} unions those layers into one set of tickers, which we treat as the reach of the simulated shock. The function \texttt{get\_pivot\_candidates(start\_ticker)} then looks at each ticker in that set: it counts how many distinct GICS sectors appear among that ticker and its graph neighbours that also lie in the set, and it keeps tickers where that count is greater than one. The return value is a list of \texttt{(ticker, sector\_diversity)} pairs sorted so that the largest diversity appears first. This is a graph-based heuristic for stocks that bridge sectors under the simulation; it is not a ready-made list of pivot stocks taken from outside our program.


\section*{Visualization Overview}
Our visualization is primarily created through the \texttt{plotly.graph\_objects}, with the positioning of graph nodes controlled by \texttt{networkx}.


Our visualization used many aspects such as color and node size to improve viewer clarity. The edge traces are built such that positive correlations create green edges, while red edges indicate that the edge represents a negative correlation. The node sizes are normalized such that a node with a higher degree would appear relatively larger in size than another node with a lower degree. The nodes themselves are generally colored by sector. Additionally, when searching for pivot candidates from a start node, the colors change so that the start ticker and identified candidates are highlighted in different colours, while the rest of the nodes are greyed out.\\

Node and edge traces are built into \texttt{go.Scatter} objects, which are then built into a \texttt{go.Figure} object to be sent to the Plotly \texttt{dash} app. We used the \texttt{dash} library to build a Flask app around our go.Figure, since dash includes built-in core components to make inputs. This improves the user interface and allows for easier user control of our interactive graph. Our interface includes a slider to control the threshold, a radio menu to control the view options, and a drop-down search to search for pivot candidates from a start ticker. These inputs interface with the elements they update through functions that have the \texttt{@callback decorator}. These decorators mean that the function runs whenever the input is used\\

To reduce the time it takes to change between different figures for different thresholds or view options, each combination of threshold and view option is precomputed. These figures are computed at the beginning of the pipeline and are stored in \texttt{figures\_by\_threshold}. This is a nested dictionary with the outer dictionary mapping the three view options of 'Standard', 'Sector', and 'Community' to an inner dictionary that maps the threshold to the figure. This allows each figure to be quickly accessed. Thus, each time the threshold slider or view option is changed, we can update the figure by returning the value from our dictionary rather than going through the process of creating each figure every time. However, this approach is not feasible when we look for pivot candidates by starter ticker, as there are a large number of possible combinations. Thus, we build the traces for the figure each time we search for pivot candidates.


We split our visualization into 3 view options: \textbf{Standard, Sector, and Community}.

Standard view uses \texttt{NetworkX.spring\_layout}, which "[treats] edges as springs holding nodes close, while treating nodes as repelling objects"\textsuperscript{9}. This visually groups stocks into clusters where nodes with edge connections to one another would be closer together, and nodes that did not have connections would be repelled outwards to the edges. However, this did not create a clear viewing experience because most of the nodes clustered into 1 large group with only a few small groups and standalone nodes on the outside.

Sector view groups the nodes into clusters based on their GICS sector through a clever layout trick taken from the NetworkX documentation.\textsuperscript{10}. We create a \texttt{nx.cycle\_graph} with length equal to the number of sectors. This creates a circle-like graph in which all the nodes are contained in a singular cycle. The positions of these nodes are then used as the centers for spring layouts of subgraphs representing each sector. Thus, all nodes of the same sector are grouped in close proximity.

Community view uses \texttt{nx.greedy\_modularity\_communities} to group the stocks into graph communities, where the number of communities matches the number of GICS sectors in the graph. Communities are often defined as groups of vertices where the density in that group is higher than the average density in the graph. \textsuperscript{11}. Community view takes the same trick from Sector view, grouping the stocks proximally into the clusters that the NetworkX algorithm finds. Essentially, if you asked the algorithm to group the full stock dataset into 11 sectors instead of the 11 GIC sectors present, this view shows how it would form groupings. \\

At the bottom of our page, we also have panels showing useful statistics regarding the graph density and the average absolute Pearson coefficient of the sector subgraphs. Under the headers for these panels, we show the values for the full graph. Further underneath, we have the values for the sectors ranked from greatest to least, so that the viewer can compare the density and correlation metrics between sectors to judge relative accuracy in the groupings. Note that density updates with the threshold changes, but the average absolute Pearson coefficient of the sectors does not because it can only be done on a complete graph.

\section*{Running the Program}
The provided file \texttt{requirements.txt} provides all the required modules to run the program. Note that when installing \texttt{yfinance-cache}, an underscore must be used instead of a hyphen, i.e. \texttt{pip install yfinance\_cache}. You can run \texttt{pip install -r requirements.txt} to install on the necessary packages.\\
Because we are retrieving data through the Yahoo Finance API, this project does not rely on a static dataset, as mentioned previously. However, the project uses the S\&P 100 tickers (as of 2024-2025) and their price data from March 2nd, 2026 to March 31st, 2026 as the project's dataset, provided in \texttt{py-yfinance-cache.zip}. The data must be extracted into \texttt{C:\textbackslash Users\textbackslash <Username>\textbackslash AppData\textbackslash Local\textbackslash yfinance\_cache} for the data to load in the program. When \texttt{main.py} is run, it takes the tickers from \texttt{constants.py} and retrieves the corresponding historical price data directly from the Yahoo Finance API. If the data already exists in the zip file, then it will be retrieved from there, significantly decreasing the retrieval time.\\
In the unlikely chance that the download fails, in the function call on line 36 of \texttt{main.py}, please replace the arguments with \texttt{tickers, thresholds, date\_range=('2026-03-02', '2026-04-01'), interval='1d'} to retrieve exclusively the data included in the zip file.\\
When \texttt{main.py} is run and you are not using data from the provided zip file, you may notice that the result will return slowly, which is due to the nature of \texttt{yfinance\_cache.download()}. With a larger sample, it will take some time to download each ticker's data before creating the graph.\\
The program visualization requires self-hosting. When the \texttt{main.py} is run, please follow the hyperlink that shows up after some time. This will lead to the opening of the graph visualization.\\

\section*{Changes from Proposal}
A few changes were made from our original proposal:
\begin{itemize}
    \item Generalized the problem statement. Initially, the project aimed to analyze sector distinctness during periods of high volatility. Upon further discussion, the group concluded that, ideally, the graph would provide meaningful results during any economic period.
    \item Implementing a mode that switches clustering in the graph from the number of edges to grouping the nodes by sector. This was inspired by our TA and provides a different perspective in the analysis.
    \item A slider in the visualization that changes the correlation threshold, influencing the results. This change was also inspired by our TA and made narrowing down very strong correlations very easy without running the program multiple times.
    \item Instead of using our Breadth-First Search algorithm to simulate a stock crash, the project used it instead to find stocks that have a strong correlation across multiple GICS sectors.
    \item The additional data upon hovering over each node consists of only the ticker name, sector, and number of neighbours. Furthermore, a leaderboard was added to showcase the five tickers with the most neighbours. Our group decided that data such as the log return was irrelevant to the representation when the graph is focused solely on correlations, so we aimed to keep the data informative yet presentable.
    \item Changing the size of a node based on the number of correlations. This way, the graph felt more dynamic and it was easier to pinpoint "important" nodes within the network.
\end{itemize}

\section*{Discussion}
This project set out to determine whether or not GICS-defined sectors accurately reflect the behavior of stocks measured by price correlation. To investigate, a correlation graph was constructed with the S\&P-100 as the ticker set, with one month of historical prices retrieved at daily intervals.\\\\

At low correlation thresholds (0.5-0.6), results showed many edges within and across sectors, suggesting a relatively connected stock market, and that a node being a part of a sector is not the sole factor that determines its price behaviour. In other words, certain stocks across sectors share a strong enough price behaviour correlation that is not reflected by the standards of GICS sectors.\\\\

As the threshold increases (0.7 - 0.8), the network becomes much sparser and more interpretable. Many weak cross-sector correlations disappeared, while stronger correlations remained intact; the same pattern observed at lower thresholds, however, remained. Some stocks maintained strong connections across sectors as seen through the sector-organised layout, while others fragmented or kept only a few strong links. This helps answer our research question: traditional sectors do retain some distinctness, but that distinctness becomes more visible only when correlations are filtered out. At lower thresholds, shared connections across the market blur lines between the GICS structure.\\\\

For example, at a threshold of 0.8, the full graph is still extremely sparse with very low density, while most stocks are isolated or only weakly connected, a select few stocks remain meaningfully connected. Furthermore, when using the “pivot-candidate” search on many of the remaining stocks, we notice that some of them still appear to influence the structure of the network beyond its own sector, even when there is no direct correlation edge remaining between them and certain other stocks.\\\\

We had implemented a threshold slider that goes from 0.1 to 1.0, but we discovered that only the thresholds mentioned above demonstrated significant results. Anything below a 0.5 threshold had too many edges and was therefore too dense to distinguish any pattern, while any threshold above 0.8 became too sparse, as there were too few edges to draw any meaningful conclusion. This made the 0.5 - 0.8 threshold range the optimal balance between visibility and interpretability.\\\\

Through pivot candidates, analyzing different thresholds, and graph node clustering by sector, we were able to identify strong correlations between sectors, implying that the GICS categorization is not absolute.\\\\

While the project was able to adequately answer our inquiry, it was not without limitations. Our first problem was with the nature of the Yahoo Finance API: downloading too much data too quickly from Yahoo Finance at once causes the site to rate-limit and block the IP address. This is what ultimately led us to use our project’s dataset, which is the S\&P 100 (from 2024-2025), a mere 100 stocks out of the tens of thousands of stocks on Yahoo Finance. We also only retrieved data over one month, which is insufficient in large scale analysis. Despite this, the runtime of the program was still noticeably slow; with a larger dataset and a longer time period, the runtime would exponentially increase. Some of our functions, like the calculation for the average absolute value of Pearson coefficients in a graph run in O(N\^2), meaning that our program would only run at reasonable speeds for up to about 10,000 input stocks. Above all, our graph is only able to analyze stocks by their Pearson correlation coefficients, which represent linear correlations, effectively overshadowing non-linear correlations between stocks. \\\\

Overall, the results were able to answer our question: Many edges appear to be between sectors, which follows the principle behind GICS sectors. We firmly believe, however, that our project still has a long way to go. First, we could implement an algorithm that uses a more sophisticated measure than the Pearson coefficient in order to see a more complete picture of the correlation between certain stocks. Second, we could try to further optimize our code to be able to handle a much larger dataset over a longer time period with a reasonable runtime. While this will not solve the rate-limiting issue, an algorithm can be developed to download a certain number of ticker data over certain intervals.
